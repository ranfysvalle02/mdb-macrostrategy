Methodology & Caveats (read me)
  • What this measures: query latency, queries/sec, index build time, and recall@k against a precomputed ground truth — exact brute-force cosine top-k for vectors, a deterministic TF-IDF top-k for full-text. Both engines index byte-for-byte identical, seed-deterministic data.
  • Why recall is shown next to latency: vector search is approximate. A low ef_search / numCandidates buys speed by returning a worse answer. The only honest comparison tunes both knobs toward a comparable recall, then compares latency. Those knobs are the fairness controls — this is the search analog of the basic angle's normalized Postgres control.
  • Full-text is a lexical-overlap signal, not a relevance verdict: Postgres ts_rank_cd and Atlas Search BM25 rank differently, so expect high-but-not-perfect overlap with the shared lexical reference. Both run OR-semantics over a non-stemming analyzer so the comparison is apples-to-apples on tokens.
  • The honest framing: both engines do search on the same record — this is not unique to MongoDB. Postgres keeps its tsvector/pgvector indexes transactionally in-line; Atlas Search/Vector run on a separate mongot process and are eventually consistent with mongod. MongoDB's edge here is the unified managed platform and scale-out, not an exclusive capability. The mongot index byte-size is not exposed by Atlas Local, so index-size is reported one-sided.
  • Scope: one workload on a single machine via Docker. Your numbers will differ; that is the point — run it yourself.
All figures are measured locally — nothing here is hand-written.
